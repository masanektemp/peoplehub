3. AUDIT LOG — WAJIB MASUK PHASE 2a (KEPUTUSAN 17 Jul 2026)
================================================================================

TUJUAN
  Trace tindakan USER dalam sistem HR (bukan ubah fail kod projek).
  Siapa buat apa, bila, pada rekod/modul mana.

BUKAN SCOPE AUDIT APP
  • Ubah fail .ts / .json dalam folder projek → itu GIT (developer)
  • Log Cursor / Daily Task.txt → log kerja dev, bukan user app

JADUAL: audit_log (cadangan)
  id            — UUID / auto increment
  created_at    — datetime (UTC atau office TZ)
  user_id       — siapa
  username      — snapshot nama login
  action        — login | logout | create | update | delete | view | export
  module        — users | employees | leave | security | ...
  target_id     — ID rekod terjejas (optional)
  target_label  — ringkasan (cth. "EMP001", "john.doe")
  details       — JSON: field berubah, nilai lama/baru (optional)
  ip_address    — dari request header
  user_agent    — browser (optional)

PERISTIWA WAJIB REKOD (MINIMUM 2a)
  □ Login berjaya / gagal
  □ Logout
  □ Create / update / delete user (/users)
  □ Reset password / role change
  □ (2b) Create / update employee

PERISTIWA ELok (2b+)
  □ Approve / reject leave, claim, OT
  □ Export laporan / cetak PDF
  □ Akses modul sensitif (payroll, medical)

API
  POST internal sahaja — src/lib/audit/log-event.ts
  GET  /api/audit       — admin sahaja, pagination + filter

UI
  Modul /security — ganti mock table dengan data audit_log sebenar
  Filter: user, modul, tarikh, action
  Admin only

PATTERN RUJUKAN
  Sistem Depoh: audit-db.js + jadual audit_log (D1)
  PeopleHub: sama konsep, guna SQL Server + Prisma

KEKALAN DATA
